## Findings

### High: HubEvent Contract Is Not Canonical Enough

The reviewed docs do not yet define one authoritative HubEvent contract for v0.6.

Observed drift:

- `kind`, `status`, `title`, `subtitle`, `priority`, `updatedAt`
- `id`, `type`, `source`, `createdAt`, `expiresAt`, `progress`, `payload`

v0.6 implementation should not start until one field set is chosen and documented as the canonical contract.

### High: Runtime Path Terminology Is Split

The docs currently describe multiple runtime paths:

- `Provider -> Event Bus -> Store -> Resolver -> UI`
- `Mock Provider -> provider adapter -> publishHubEvent() -> Store -> Resolver -> UI`
- `ProviderRegistry -> Provider -> HubEvent -> Store`

The expected canonical path should be clarified before implementation, likely:

```text
ProviderRegistry
  -> Provider
  -> provider adapter
  -> Event Bus / publishHubEvent()
  -> Store
  -> Resolver
  -> UI
```

### High: Lifecycle, Health, And Event Status Are Conflated

Provider lifecycle state, registry health state, and event/task status are not consistently separated.

Observed lifecycle vocabularies include:

- `idle`, `starting`, `running`, `stopping`, `stopped`, `error`
- `Registered`, `Started`, `Publishing Events`, `Paused`, `Stopped`, `Error`
- `stopped`, `starting`, `running`, `pausing`, `paused`, `stopping`, `failed`, `unavailable`

v0.6 needs one provider lifecycle model, one registry health model, and one event/task status model.

### High: Provider Contract And Registry Behavior Are Not Fully Frozen

`pause()` and `resume()` appear in lifecycle or registry docs, but are not consistently represented in the base provider contract.

Registry and runtime ownership is also blurred: both appear to own start/stop/select/health transitions. The ownership boundary must be settled before implementation.

### High: Provider-To-HubEvent Boundary Is Inconsistent

Some docs say providers emit `HubEvent[]` directly. Other docs imply runtime converts provider output into HubEvents.

The existing source direction appears simpler:

- `HubProviderStatus = "stopped" | "running" | "error"`
- mock providers emit `HubEvent[]` through an adapter

The docs should decide whether v0.6 follows the existing adapter path or introduces a runtime conversion layer.

### High: v0.5 / v0.6 Scope Is Contradictory

`docs/IMPLEMENTATION_PLAN.md` references provider source and test files while also stating that v0.5.1 has code changes `0` and must not implement provider runtime, mock providers, adapters, registry, or tests.

v0.6 needs a clear baseline statement:

- what already exists
- what is planned
- what is explicitly out of scope

### High: Per-Provider Registry Matrix Is Missing

The four planned mock providers do not yet have implementation-ready registry expectations:

- stable provider ids
- provider groups
- capabilities
- metadata
- initial lifecycle state
- registration order

Affected providers:

- `MockMusicProvider`
- `MockAIProvider`
- `MockDownloadProvider`
- `MockNotificationProvider`

## Non-Blocking Notes

- Lifecycle coverage is strong at the concept level: start, stop, duplicate start, unsubscribe, cleanup, deterministic timers, stale event prevention, and error events are documented across the planning docs.
- Failure handling is mostly sufficient at policy level: concise error events, no sensitive payloads, failed start/stop assertions, provider health, and crash isolation are present.
- The test strategy is useful but should become a v0.6 acceptance matrix before implementation.
- AI provider naming drift is medium severity, not itself a blocker if fixed during the alignment pass.

## Open Questions

1. Should v0.6 implement all four mock providers in one slice, or start with one provider plus shared tests?
2. What is the canonical AI provider name: `MockAIProvider`, `MockAITaskProvider`, or `MockAiTaskProvider`?
3. Is a registry implementation required in v0.6, or is adapter-only acceptable for the first runtime slice?
4. Does the canonical provider contract include `pause()` and `resume()`, or only `start()` and `stop()` for the first slice?
5. Should providers emit `HubEvent[]` directly, or should runtime convert provider-specific output into HubEvents?

## Freeze Decision

**Rework**

The current docs are close, but they are not yet frozen as implementation-ready. v0.6 Mock Provider SDK Implementation is **not authorized** until the high-severity blockers above are resolved by a focused documentation alignment pass.

Recommended next phase:

```text
v0.5.3 Documentation Alignment
```

Suggested alignment scope:

- canonical HubEvent contract
- canonical runtime path
- provider lifecycle vs registry health vs event status vocabulary
- provider contract including or excluding pause/resume
- provider-to-HubEvent boundary
- per-provider registry matrix
- v0.6 implementation slice definition
