# v0.8 Runtime Capability Transition Evidence Plan

## Purpose

This document defines the evidence required before any later runtime capability slice may propose changing `windowsProviders: false`. It is a docs-only readiness artifact. It does not change runtime capability facts, implement provider behavior, call a Windows API, observe the operating system, create a provider export path, emit events, start polling, change lifecycle behavior, or wire runtime capabilities into provider behavior.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Transition Rule

`windowsProviders` may change only in a later, separately approved implementation slice after the evidence below is reviewed. Until then, runtime capability loading must continue to treat `windowsProviders: false` as the truthful capability fact.

This plan does not propose the changed value. It defines the review gate that must exist before a later slice can propose one.

## Evidence Required Before A Future Proposal

| Evidence | Required proof | Boundary |
| --- | --- | --- |
| Privacy checklist satisfied | The system status privacy checklist has approved allowed coarse facts and forbidden private facts. | No raw process lists, window titles, command output, usernames, paths, app identities, raw network details, secrets, or stable identifiers. |
| Descriptor boundary satisfied | The system status preflight descriptor remains a diagnostic fact with `id: "system-status"`, `kind: "system-status"`, `origin: "native"`, and `support: "preflight"`. | The descriptor is not a provider export path, provider registration, runtime-provider wiring, or lifecycle state. |
| Redacted diagnostics proven | Unsupported platform, source unavailable, permission denied, malformed source data, timeout, and provider bug cases have bounded diagnostic shapes. | Diagnostics use short codes, booleans, and coarse enums without private data or raw source payloads. |
| Native boundary design reviewed | A future native boundary design explains what source would be queried, what permission model applies, and how failures stay local to diagnostics. | No Windows API, native handle, OS observation, polling implementation, background process, or Tauri/Rust command wiring is approved by this document. |
| Failure modes defined | Unsupported platform, unavailable source, permission denied, malformed source data, timeout, and provider bug behavior are named before source behavior is added. | Failure handling must not directly mutate Store, Resolver, UI, or provider lifecycle behavior. |
| Runtime payload contract planned | A future runtime capability payload contract defines how a capability fact crosses the `get_runtime_capabilities` boundary. | The contract must remain separate from HubEvent payloads, event emission, provider lifecycle, and provider availability. |
| Separate implementation authorization required | Commander must issue a later explicit implementation-only authorization for any native, runtime, provider, Tauri/Rust, or code change. | This plan authorizes no implementation and no `windowsProviders` change. |

## Runtime Capability Boundary

Runtime capability facts describe what the runtime can truthfully report. They are separate from:

- Provider availability.
- Provider readiness.
- Provider connection state.
- Provider activity.
- Provider implementation state.
- Provider lifecycle.
- Event emission.
- UI behavior.

A future runtime capability transition must not treat `origin: "native"` or `support: "preflight"` as proof that a provider exists or that native system status behavior is available. Those fields remain diagnostic metadata until a separately reviewed implementation proves otherwise.

## Required Future Proposal Shape

Any later slice that proposes a `windowsProviders` transition should include:

1. The exact runtime capability field change being proposed.
2. The evidence package from this document.
3. The source boundary and permission model.
4. The redacted diagnostic behavior for each failure category.
5. The runtime payload contract for `get_runtime_capabilities`.
6. The tests required to prove malformed, unavailable, denied, timeout, and provider bug paths stay bounded.
7. Explicit stop conditions for provider exports, runtime-provider wiring, event emission, polling implementation, native handles, and lifecycle behavior.

If any item is missing, the future slice should keep `windowsProviders: false` and re-plan.

## Non-Goals

This document does not:

- Change `windowsProviders: false`.
- Add runtime, provider, Tauri/Rust, config, package, dependency, script, Store, Resolver, event bus, UI, or showcase changes.
- Add Windows API calls, native handles, OS observation, background processes, polling implementation, provider export paths, runtime-provider wiring, event emission, or lifecycle behavior.
- Claim native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Review Criteria

A future runtime capability transition plan should be accepted only if it:

- Keeps the current `windowsProviders: false` fact truthful until a later approved implementation changes it.
- Keeps `origin: "native"` and `support: "preflight"` diagnostic-only.
- Separates runtime capability facts from provider availability, provider lifecycle, provider implementation, event emission, and UI behavior.
- Shows that privacy, descriptor, diagnostics, native boundary, failure modes, and runtime payload contract evidence have all been reviewed.
- Requires a separate implementation authorization before any native/runtime/provider work begins.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Code, tests, config, package, dependency, script, Rust, Tauri, runtime, provider, store, resolver, event bus, UI, or showcase changes outside its approved scope.
- Windows API calls, native handles, OS observation, polling implementation, or background processes.
- Provider export paths, real provider files, provider startup behavior, provider subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring.
- A `windowsProviders` value change without the evidence package defined here.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Next Sequencing

The next safe follow-up remains non-native unless Commander explicitly approves a narrower code or test scope. Prefer either a docs-only native boundary design or a tests-only runtime payload contract using local fixtures. Real system status provider work and any runtime capability transition should wait until those gates are separately reviewed and approved.
