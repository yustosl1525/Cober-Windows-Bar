# v0.8 System Status Preflight Descriptor Plan

## Purpose

This document defines the descriptor boundary for a future system status capability fact. It is a docs-only readiness artifact. It does not implement provider behavior, call a Windows API, observe the operating system, start polling, create a provider export path, emit events, change lifecycle behavior, or wire runtime capabilities into provider behavior.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Descriptor Intent

A system status preflight descriptor is diagnostic metadata about a possible future capability. It is not a provider object, not a provider registration, not runtime capability enablement, and not evidence that the operating system has been observed.

The descriptor may answer only this question:

```text
Can the diagnostic model name a future system status capability without creating provider behavior?
```

It must not answer whether a native source exists, whether a provider can start, whether data can be polled, or whether events can be emitted.

## Allowed Descriptor Facts

The future descriptor may use these facts only if a later approved slice needs a local fixture or documentation example:

| Fact | Allowed value | Rule |
| --- | --- | --- |
| `id` | `"system-status"` | Capability id only, not a provider id or export path. |
| `kind` | `"system-status"` | Capability family only, not an active source. |
| `origin` | `"native"` | Future-facing origin label only. |
| `support` | `"preflight"` | Diagnostic support label only. |
| `surface` | `"systemStatusPreflight"` | Optional diagnostic surface name for local fixtures. |
| `windowsProviders` | `false` | Runtime capability truth remains unchanged. |
| `diagnosticCategories` | Short category codes listed below. | Diagnostic categories only, not source behavior. |

Allowed diagnostic category codes are the Slice 29-A local fixture categories:

```text
unsupported
unavailable
permission-denied
malformed
timeout
invoke-failed
```

The descriptor may reference these categories to describe bounded diagnostics. It must not expand them into native calls, source handles, payload schemas, event publishing rules, or retry policy.

## Forbidden Descriptor Fields

The descriptor must not include fields that imply provider behavior, runtime enablement, private data, or source observation.

Forbidden fields and meanings include:

| Field or concept | Reason |
| --- | --- |
| `provider`, `providerId`, `providerExport`, or module path | Would imply a provider export or implementation path. |
| `available`, `ready`, `connected`, `active`, `implemented`, or `productionCapable` | Would imply native capability state that has not been proven. |
| `enabled`, `registered`, `started`, `start`, `stop`, `subscribe`, or `lifecycle` | Would cross into provider lifecycle behavior. |
| `emit`, `event`, `eventBus`, `payload`, `store`, `resolver`, or `ui` | Would cross into event emission or application state flow. |
| `runtimeProviderWiring` or runtime capability transition flags | Would imply runtime-provider wiring or `windowsProviders` changes. |
| `windowsApi`, `nativeHandle`, `osObservation`, `polling`, or `backgroundProcess` | Would imply native calls, OS observation, or background behavior. |
| process, window, command, username, path, app identity, raw network, secret, or identifier fields | Would violate the system status privacy checklist. |

If a later slice needs any forbidden field, it must stop and re-plan before implementation.

## Relationship To Existing Readiness Artifacts

The system status privacy checklist defines which coarse facts may be considered later and which facts are forbidden. This descriptor plan does not add new allowed data. It only names the small diagnostic metadata shape that may describe the future capability.

Slice 29-A tests prove local diagnostic result shapes for unsupported, unavailable, permission denied, malformed, timeout, and invoke failed cases. This descriptor plan does not add tests, fixtures, code, or production behavior. It keeps those categories as short diagnostic labels.

## Descriptor Versus Provider Behavior

| Boundary | Descriptor may say | Descriptor must not do |
| --- | --- | --- |
| Capability identity | The future capability id is `system-status`. | Create a provider file, provider export path, or provider id. |
| Capability origin | The diagnostic origin label is `native`. | Claim a native source has been observed. |
| Capability support | The diagnostic support label is `preflight`. | Claim native support is available, ready, connected, active, implemented, or production-capable. |
| Runtime capabilities | `windowsProviders: false` remains truthful. | Enable or imply a runtime capability transition. |
| Diagnostics | Reference short category codes. | Emit events, publish HubEvent payloads, or define UI behavior. |
| Privacy | Defer to the privacy checklist. | Carry raw OS data, command output, paths, identifiers, or private content. |

## Review Criteria

A future descriptor or fixture slice should be accepted only if it:

- Keeps `windowsProviders: false` truthful.
- Keeps `origin: "native"` and `support: "preflight"` diagnostic-only.
- Uses `id: "system-status"` and `kind: "system-status"` as capability facts only.
- Reuses the existing coarse diagnostic categories without expanding source behavior.
- Avoids provider lifecycle, event emission, runtime-provider wiring, polling implementation, native calls, and provider export paths.
- Avoids wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Code, tests, config, package, dependency, script, Rust, Tauri, runtime, provider, store, resolver, event bus, UI, or showcase changes outside its approved scope.
- Windows API calls, native handles, OS observation, polling implementation, or background processes.
- Provider export paths, real provider files, provider startup behavior, provider subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring or a runtime capability transition.
- New private data allowances beyond the system status privacy checklist.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Next Sequencing

The next safe follow-up after this descriptor plan is still non-native. Prefer a docs-only runtime capability transition plan that defines what evidence would be required before `windowsProviders` could truthfully change. Real system status provider implementation should wait until descriptor, privacy, diagnostic shape, and runtime capability transition readiness have each been separately reviewed and approved.
