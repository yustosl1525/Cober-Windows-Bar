# v0.8 System Status Runtime Payload Contract Plan

## Purpose

This document defines the future runtime payload contract boundary for system status capability facts crossing the `get_runtime_capabilities` boundary. It is a docs-only readiness artifact. It does not change runtime code, provider code, tests, Tauri/Rust wiring, native collection, event emission, runtime-provider wiring, UI exposure, or `windowsProviders: false`.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Contract Intent

A future system status runtime payload contract may describe how coarse, redacted facts could be represented after the privacy, descriptor, diagnostic, transition evidence, and native boundary gates are reviewed. It must not describe source collection, provider lifecycle, provider availability, event emission, Store or Resolver behavior, UI behavior, or a `windowsProviders` transition.

The planned contract question is:

```text
Which bounded facts may a future runtime capability payload name without exposing private data or implying provider behavior?
```

This slice does not answer whether a native source exists, whether a provider can run, or whether runtime capability values should change.

## Allowed Future Coarse Payload Facts

Future payload planning may discuss only derived, coarse system status facts already allowed by the privacy checklist:

| Fact | Allowed shape | Contract rule |
| --- | --- | --- |
| CPU range | `low`, `medium`, `high`, `critical`, or `unknown` | Derived load category only. No process, thread, command, window, app, or per-core detail. |
| Memory pressure | `normal`, `elevated`, `high`, `critical`, or `unknown` | Derived pressure category only. No process lists, memory maps, usernames, paths, or app identities. |
| Battery state | `charging`, `discharging`, `full`, `low`, `critical`, or `unknown` | Derived state only. No device serials, account identifiers, hardware ids, or power-plan names. |
| Network availability summary | `offline`, `online`, `limited`, `metered`, or `unknown` | Derived availability category only. No SSIDs, IP addresses, MAC addresses, hostnames, endpoints, packet data, DNS queries, or traffic content. |

These facts remain future contract candidates only. They do not change `get_runtime_capabilities`, `windowsProviders`, provider descriptors, HubEvents, Store state, Resolver output, or UI.

## Forbidden Payload Content

The runtime payload contract must not include raw or private content.

Forbidden content includes:

- Secrets, tokens, credentials, cookies, environment variables, or config values.
- File paths, folder paths, project paths, downloads paths, home directories, drive contents, or registry paths.
- Command output, command lines, stdout, stderr, shell text, stack dumps with paths, or raw source payloads.
- Usernames, account names, user identifiers, policy details, or permission prompt text.
- App identities, package identities, foreground app names, app usage history, process names, process ids, or raw process lists.
- Window titles, notification text, notification metadata, transcript content, prompts, chat logs, or generated text.
- Raw network data, SSIDs, IP addresses, MAC addresses, hostnames, remote endpoints, packet data, DNS queries, or traffic contents.
- Stable identifiers, machine identifiers, device identifiers, hardware serial numbers, source handles, or native handles.
- Raw OS data or environment dumps.

If a future payload proposal needs any forbidden content, it must stop and re-plan before implementation.

## Diagnostic And Error Summaries

Runtime payload planning may reference short diagnostic categories only:

```text
unsupported
unavailable
permission-denied
malformed
timeout
invoke-failed
```

Diagnostic summaries must stay bounded, redacted, and facts-only. They may use short codes, booleans, and coarse enum values. They must not include raw source responses, private content, command output, paths, identifiers, provider lifecycle state, runtime-provider wiring state, event emission state, or UI state.

## Separation Rules

### Runtime Payload Versus Provider Availability

A runtime payload fact must not say that a provider is available, ready, connected, active, implemented, production-capable, or already wired. Runtime payload facts describe possible capability evidence only after separate review.

### Runtime Payload Versus Provider Lifecycle

Provider lifecycle remains out of scope. A runtime payload fact must not imply a provider is registered, started, stopped, subscribed, publishing, failed, or otherwise operating.

### Runtime Payload Versus HubEvents

This contract does not authorize HubEvent payloads. Coarse system status facts must not become HubEvents, Store mutations, Resolver behavior, event bus behavior, or UI behavior without a later approved slice.

### Runtime Payload Versus Native Collection

This contract does not authorize native collection. It must not call a Windows API, observe the operating system, create native handles, start polling, run a background process, or wire Tauri/Rust commands.

### Runtime Payload Versus Runtime-Provider Wiring

This contract does not authorize runtime-provider wiring. A future runtime payload must remain separate from provider exports, provider registrations, provider adapters, and provider lifecycle behavior.

## Relationship To Existing Gates

This contract preserves the existing v0.8 readiness gates:

- The privacy checklist owns allowed coarse facts and forbidden private data.
- The preflight descriptor plan owns `origin: "native"` and `support: "preflight"` as diagnostic-only capability metadata.
- The Slice 29-A diagnostic categories remain local fixture categories until a later slice proves otherwise.
- The runtime capability transition evidence plan owns the evidence required before any `windowsProviders` proposal.
- The native boundary design plan owns future collection, redaction, runtime payload planning, provider lifecycle, event emission, and UI exposure separation.

This document does not replace those gates. It narrows the runtime payload contract vocabulary so later tests can use local fixtures without implementing runtime or provider behavior.

## Review Criteria

A future runtime payload proposal should be accepted only if it:

- Keeps `windowsProviders: false` truthful and unchanged until a later approved runtime capability implementation changes it.
- Keeps `origin: "native"` and `support: "preflight"` diagnostic-only.
- Uses only allowed coarse facts and redacted diagnostic categories.
- Excludes secrets, paths, command output, usernames, app identities, raw network data, stable identifiers, raw OS data, and private content.
- Separates runtime payload facts from provider availability, provider lifecycle, HubEvents, Store, Resolver, UI, native collection, event emission, and runtime-provider wiring.
- Requires separate implementation authorization before any runtime, provider, native, Tauri/Rust, event, or UI work begins.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Code, tests, config, package, dependency, script, Rust, Tauri, runtime, provider, store, resolver, event bus, UI, or showcase changes outside its approved scope.
- Windows API calls, native handles, OS observation, polling implementation, or background processes.
- Provider export paths, real provider files, provider startup behavior, provider subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring or a `windowsProviders` value change.
- Raw or private payload content.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Next Sequencing

The next safe follow-up remains non-native. Prefer tests-only local fixture coverage for this planned runtime payload shape, with no runtime code changes and no provider behavior. Real system status provider work and any runtime capability transition should wait until privacy, descriptor, diagnostics, transition evidence, native boundary, runtime payload, and fixture gates are each separately reviewed and approved.
