# v0.8 First Provider Candidate Decision

## Decision

Choose **system status** as the first v0.8 provider candidate for readiness sequencing.

This is a planning decision only. It does not implement a provider, call a Windows API, add Tauri/Rust command wiring, create a provider export path, observe the operating system, start a background process, or change runtime capability facts.

The current v0.7 baseline remains truthful:

- `windowsProviders: false` remains the runtime capability fact.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- No native Windows, Music, system status, or Downloads provider is available, ready, connected, active, implemented, production-capable, or already wired.

## Candidates Compared

| Candidate | Privacy risk | Permission/config/dependency risk | Runtime/provider wiring risk | Diagnostic value | Product value | Decision |
| --- | --- | --- | --- | --- | --- | --- |
| System status | Lowest if limited to coarse CPU, RAM, battery, or network summary facts and no raw process/window data. | Lowest for planning; future implementation may still need Windows API and Tauri permission review. | Keep at zero for this slice; future probes must not wire runtime to provider behavior early. | High: exercises unsupported source, unavailable source, stale data, and bounded polling diagnostics. | Medium/high: useful always-on status surface with low content sensitivity. | Select first. |
| Music capability | Medium: track title, artist, album art, app identity, and playback state can expose private listening context. | Medium/high: future Windows Media or GSMTC design needs API compatibility and privacy review. | Higher because it tempts direct mapping from native media session facts into existing Music UI. | High, but only after metadata privacy and source compatibility are planned. | High: visible user-facing value and existing mock Music mode. | Defer. |
| Downloads | Medium/high: filenames, paths, partial files, browser-specific state, and download history can expose private data. | Medium/high: future watcher behavior, file-system access, and browser integration choices need review. | Higher because file watchers or background observation can expand beyond diagnostics. | Medium: useful for unavailable source and path-redaction diagnostics. | Medium: maps to existing Download mode but has path privacy risk. | Defer. |

## Why System Status Goes First

System status is the safest first candidate because it can be scoped to derived, coarse facts instead of private content. A future design can begin with summaries such as CPU load, memory pressure, battery state, or network availability while explicitly excluding raw process lists, window titles, command output, usernames, paths, app-specific private data, and notification or transcript content.

It also has strong diagnostic value. The first real provider lane needs to prove how the project handles unsupported platform, unavailable source, stale data, timeout, permission denied, malformed source data, and provider bug cases before any source data is treated as product behavior. System status gives those diagnostics without immediately taking on the metadata and compatibility risk of Music or the file path risk of Downloads.

## Why Music Is Deferred

Music remains a high-product-value candidate, but it should wait until a specific privacy and metadata boundary exists.

Deferral reasons:

- A future Windows Media or GSMTC plan must define what metadata may cross into `HubEvent.payload` or `HubEvent.metadata`.
- Track title, artist, album art, source app identity, playback position, and playback state need explicit privacy review.
- The existing `origin: "native"` / `support: "preflight"` music fact is a diagnostic descriptor only.
- No future Music slice should claim native support is available, ready, connected, active, implemented, production-capable, or already wired before a reviewed proof exists.

## Why Downloads Are Deferred

Downloads are useful, but they need a file privacy design before any provider implementation.

Deferral reasons:

- Filenames, full paths, browser-specific data, partial-download conventions, and download history can expose private content.
- Watcher behavior can create background-process and OS-observation scope creep.
- A future Downloads plan must define whether the provider emits derived progress only, redacted filenames, or no file identity at all.
- Any future file watcher, native handle, background process, or OS observation must be separately planned and approved.

## Acceptance Criteria For This Decision Record

This document is complete when it:

- Names system status as the first v0.8 provider candidate.
- Compares system status, music capability, and downloads across privacy, permission/config/dependency, runtime/provider wiring, diagnostic value, and product value.
- Explains why music and downloads are deferred.
- Repeats the `windowsProviders: false` and `origin: "native"` / `support: "preflight"` diagnostic-only baseline.
- Lists safe follow-up slices that remain docs-only or tests-only until a separate native implementation gate exists.
- Repeats stop conditions for Windows API usage, GSMTC, Windows Media, provider exports, background processes, and runtime-provider wiring.

## Safe Follow-Up Slices

Prefer these before any real provider implementation:

1. **System status privacy checklist** - docs-only artifact defining exactly which coarse facts may be read, which facts are forbidden, and how errors are redacted.
2. **System status diagnostic result shape tests** - tests-only slice using local fixtures to prove unsupported, unavailable, permission denied, malformed, timeout, and provider bug diagnostics can be represented without native calls.
3. **System status preflight descriptor plan** - docs-only or tests-only slice that describes a future capability descriptor without creating a provider export path.
4. **Runtime capability transition plan** - docs-only slice that defines what evidence would be needed before `windowsProviders` could truthfully change in a later runtime capability slice.

Real provider implementation should wait until these readiness checks are reviewed and approved.

## Stop Conditions

Stop and re-plan if a proposed Slice 27-A follow-up needs:

- Windows API, GSMTC, Windows Media, native handles, OS observation, or background processes.
- Tauri/Rust command wiring, permissions, capabilities, config, package/dependency/script changes, or production packaging.
- A provider export path, real provider file, provider startup behavior, provider subscription behavior, or event emission behavior.
- Runtime-provider wiring.
- Provider adapter, store, resolver, event bus, UI, or showcase changes.
- Any wording that implies native Windows, Music, system status, or Downloads support is available, ready, connected, active, implemented, production-capable, or already wired.

## Current Boundary

This decision record narrows sequencing only. It does not make native support available, does not change runtime capability facts, and does not convert preflight diagnostics into provider behavior.

## Readiness Slice Update

The current v0.8 readiness slice is **system status preflight descriptor + diagnostic tests**. It prepares the runtime and desktop status surface to describe source health before any real provider exists.

Allowed diagnostic fields are limited to:

| Field | Allowed values |
| --- | --- |
| `quality` | `live`, `fallback`, `stale`, `unavailable` |
| `code` | `unsupported`, `unavailable`, `permission-denied`, `malformed`, `timeout`, `invoke-failed` |
| `source` | `mock`, `tauri-fixture`, `tauri-event`, `preflight` |
| `lastSuccessfulSource` | Optional `mock`, `tauri-fixture`, `tauri-event`, or `preflight` |

This slice may normalize legacy fixture metrics and failed preflight attempts into high-level diagnostic states. It must not add a Rust metrics reader, call Windows APIs, observe the OS, add provider exports, change `HubEvent`, change provider registry semantics, or collect private system data.

The desktop UI may display only compact source-health labels such as `Live`, `Fallback`, `Stale`, or `Unavailable`. The label is diagnostic/source-health feedback, not a claim that a real Windows system status provider has shipped.
