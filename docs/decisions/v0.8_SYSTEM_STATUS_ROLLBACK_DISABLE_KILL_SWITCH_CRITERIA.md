# v0.8 System Status Rollback / Disable / Kill-Switch Criteria

## Purpose

This document defines rollback, disable, and kill-switch criteria for the future system status capability. It is a docs-only readiness artifact. It records behavior and gate-level criteria only.

This document does not create real toggles, configuration, implementation mechanics, provider lifecycle behavior, runtime-provider wiring, event emission, UI exposure, native collection, permissions, capabilities, package changes, or dependency changes.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful and unchanged.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native/runtime/provider implementation remains not yet ready.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Criteria Intent

These criteria answer this planning question:

```text
What evidence would require a future system status capability to remain disabled, roll back, or stop before native/provider/runtime work proceeds?
```

They do not answer whether a native source exists, whether a runtime payload should change, whether a provider can start, whether events can be emitted, or whether UI should expose system status facts.

## Rollback Triggers

A later system status slice should roll back to the last approved docs-only or tests-only boundary if any of these conditions appear:

- A future source design needs private or raw data to produce useful system status facts.
- Diagnostic output cannot stay bounded, redacted, and facts-only.
- Runtime payload planning needs fields outside the approved coarse vocabulary.
- A proposed `windowsProviders` transition lacks the complete evidence package from the runtime capability transition evidence plan.
- Native boundary review cannot separate source collection, redaction, diagnostics, runtime payload planning, provider lifecycle, event emission, and UI exposure.
- Any proposal requires Windows API calls, native handles, OS observation, polling, background processes, provider export paths, runtime-provider wiring, lifecycle behavior, or event emission before a separate implementation authorization.
- Any wording implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

Rollback here means returning the future slice to planning or local-fixture evidence. It does not define implementation mechanics, runtime behavior, config changes, or a real switch.

## Disable Criteria

A future system status capability should remain disabled at the readiness gate if any required evidence is missing:

| Missing evidence | Disable rule |
| --- | --- |
| Privacy checklist coverage | Keep the capability disabled if allowed coarse facts and forbidden private facts are not reviewed together. |
| Descriptor boundary | Keep the capability disabled if `origin: "native"` or `support: "preflight"` is treated as provider availability. |
| Redacted diagnostics | Keep the capability disabled if unsupported platform, source unavailable, permission denied, malformed source data, timeout, and provider bug cases are not bounded. |
| Native boundary design | Keep the capability disabled if source collection, redaction, diagnostics, runtime payload planning, provider lifecycle, event emission, or UI exposure cannot stay separated. |
| Runtime payload contract | Keep the capability disabled if payload facts need secrets, paths, command output, usernames, app identities, raw network data, stable identifiers, raw OS data, or other private content. |
| Runtime transition evidence | Keep `windowsProviders: false` if any evidence required before a future transition is missing. |
| Approval boundary | Keep the capability disabled if Commander has not issued a separate implementation authorization for the exact future scope. |

Disabled means blocked at the readiness gate. It does not create a runtime flag, config key, provider state, UI control, event path, or package change.

## Kill-Switch Decision Inputs

A future kill-switch decision should use only reviewable, non-private inputs:

- Which approved gate failed.
- Which coarse diagnostic category applies.
- Whether the failure is retryable or must block the future slice.
- Whether `windowsProviders: false` still reflects the truthful runtime capability fact.
- Whether diagnostics remained bounded, redacted, and free of private/raw data.
- Whether the proposed scope crossed into native collection, provider lifecycle, runtime-provider wiring, event emission, UI exposure, permissions, capabilities, config, package, or dependency changes.
- Which owner approved the stop, rollback, disable decision, or re-plan request.

These inputs are criteria for later review only. They do not create a real kill switch, toggle, runtime setting, provider command, event handler, UI affordance, or native control path.

## Required Evidence And Observability

Any later implementation proposal should provide evidence that reviewers can inspect without private data:

- The exact approved scope and file list.
- The current `windowsProviders: false` baseline and any proposed future change.
- The diagnostic categories that would be reported.
- The redaction proof for each diagnostic category.
- The source boundary and failure mode summary.
- The runtime payload contract shape, if a future runtime payload is proposed.
- The stop condition that would apply if the proposal exceeds the approved boundary.

Observability remains diagnostic-only. It must not add logs, telemetry, HubEvents, Store mutation, Resolver output, UI exposure, polling, native handles, OS observation, provider export paths, or runtime-provider wiring in this slice.

## Owner And Approval Boundaries

Future rollback, disable, or kill-switch decisions should be review-gated:

- Commander owns slice authorization and scope.
- Tester owns read-only review and approval status.
- Developer must stop if the approved scope would require implementation mechanics outside the current gate.
- Any native/runtime/provider implementation requires a later, separate implementation-only authorization.
- Any `windowsProviders` transition requires a later, separate runtime capability gate.

This artifact does not approve native/provider/runtime work.

## Diagnostic Reporting Rules

Future diagnostics may use only short, bounded categories already covered by local fixtures:

```text
unsupported-platform
source-unavailable
permission-denied
malformed-source-data
timeout
provider-bug
```

Diagnostics must remain redacted and facts-only. They must not include secrets, paths, command output, usernames, app identities, raw network data, stable identifiers, raw OS data, raw source payloads, provider lifecycle state, runtime-provider wiring state, event emission state, UI state, native handles, or OS observation details.

Diagnostic wording must not imply native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Review Criteria

A later rollback, disable, or kill-switch criteria review should be accepted only if it:

- Keeps `windowsProviders: false` truthful and unchanged.
- Keeps `origin: "native"` and `support: "preflight"` diagnostic-only.
- Keeps native/runtime/provider implementation not yet ready until later gates are approved.
- Defines rollback triggers without implementation mechanics.
- Defines disable criteria without creating real toggles or config.
- Defines kill-switch decision inputs without creating a real switch.
- Requires bounded, redacted diagnostics with no private/raw data.
- Separates criteria from provider lifecycle, event emission, runtime-provider wiring, native collection, UI exposure, permissions, capabilities, config, package, and dependency changes.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Code, tests, config, package, dependency, script, Rust, Tauri, runtime, provider, store, resolver, event bus, UI, or showcase changes outside its approved scope.
- Real toggles, permissions, capabilities, package changes, config changes, or dependency changes.
- Windows API calls, native handles, OS observation, polling implementation, or background processes.
- Provider export paths, real provider files, provider startup behavior, provider subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring or a `windowsProviders` value change.
- Raw or private payload content.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.
