# v0.8 First Real Provider Readiness

## Purpose

This document is a readiness checkpoint for the first real provider phase. It does not authorize provider implementation, native Windows API usage, Tauri/Rust command wiring, capability/config changes, background processes, or runtime-provider wiring.

v0.7 closed with mock/fixture runtime diagnostics and provider capability diagnostic facts. That baseline remains the source of truth until a later approved slice proves otherwise:

- `windowsProviders: false` remains truthful.
- Tray and always-on-top remain unsupported runtime capability facts.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- No native Windows or Music provider is available, ready, connected, active, implemented, or production-capable.

## Entry Criteria

Before any first real provider implementation starts, the project should have:

- A selected first provider candidate and a written reason for choosing it.
- A provider-specific privacy and permission checklist.
- A native boundary design that can fail without changing hub UI state directly.
- A diagnostic result shape for unsupported platform, unavailable source, permission denied, malformed source data, timeout, and provider bug cases.
- A test plan that proves no provider lifecycle, adapter, store, resolver, event bus, or UI behavior changes happen accidentally.
- A stop condition for any work that needs package/dependency changes, Tauri/Rust command wiring, Windows capability/config changes, or background processes.

## Candidate Ordering

### 1. System Status Candidate

System status is the lowest-privacy first candidate if it can start with coarse facts such as CPU, RAM, battery, or network summary data. It should avoid raw process lists, window titles, command output, usernames, paths, or app-specific private data.

Readiness value:

- Proves a native-ish provider boundary with lower content privacy risk.
- Exercises polling, stale-data handling, and unsupported-source diagnostics.
- Keeps the hub status-oriented without touching media-session compatibility yet.

Deferral reasons:

- Any implementation requiring new dependencies, persistent background processes, Tauri permissions, or Windows API calls must wait for a separately approved implementation slice.
- Any design that needs continuous polling policy, power behavior, or user settings belongs in a later planning slice.

### 2. Music Capability Candidate

Music is product-visible and matches the existing mock Music hub mode, but it carries higher compatibility and privacy risk. A future Windows Media or GSMTC-based design must first define what metadata may cross the boundary and what stays private.

Readiness value:

- Connects directly to an existing mock surface.
- Tests whether future native music facts can map to the current `HubEvent` shape.
- Uses the existing native/music `preflight` diagnostic language as the planning bridge.

Deferral reasons:

- No Windows Media or GSMTC call is approved by this readiness artifact.
- Track title, artist, app identity, playback state, album art, and source app compatibility need a privacy review before implementation.
- The current native/music descriptor is `preflight` only and must not be described as working provider behavior.

### 3. Downloads Candidate

Downloads are useful but need file-system privacy rules before native work starts. A future design should decide whether filenames, paths, partial files, or browser-specific download data may be observed or emitted.

Readiness value:

- Maps well to the existing mock Download hub mode.
- Can be designed around derived progress facts instead of raw paths.

Deferral reasons:

- File watcher behavior, path privacy, partial-download conventions, and browser-specific integrations should be planned before implementation.
- Any watcher, OS handle, or background process is outside this readiness slice.

## Privacy And Permission Checklist

Every future real provider candidate must answer these before implementation:

- What data is read?
- What data is never read?
- What data crosses into `HubEvent.payload` or `HubEvent.metadata`?
- What data stays inside the provider boundary?
- Can a useful status be derived without raw private content?
- What permissions, Windows capabilities, Tauri permissions, or user prompts would be needed later?
- What happens when permission is denied?
- What happens when the source is unsupported or unavailable?
- How are errors summarized without secrets, full paths, raw notification text, transcript content, credentials, or command output?

## Diagnostic Expectations

The first real provider path should begin with diagnostics before source behavior:

- Keep capability descriptors separate from lifecycle.
- Keep provider lifecycle separate from event/task status.
- Report unsupported or unavailable source facts without starting a provider.
- Preserve `windowsProviders: false` until a later runtime capability slice truthfully changes it.
- Keep `origin: "native"` and `support: "preflight"` as future-facing capability facts until implementation proves a more specific result.
- Convert source failures into bounded diagnostics, not direct UI state changes.

Suggested future diagnostic categories:

```text
unsupported-platform
source-unavailable
permission-denied
malformed-source-data
timeout
provider-bug
```

These categories are planning language only. They are not implemented by this document.

## Review Criteria

A later first-provider implementation plan should be rejected if it:

- Claims native support is available, ready, connected, active, implemented, or production-capable before a reviewed proof exists.
- Adds Windows API, Windows Media, GSMTC, Tauri/Rust command wiring, permissions, capabilities, config, package, dependency, or script changes without explicit scope.
- Creates a provider export/path/background process before diagnostics and privacy review are approved.
- Wires runtime capabilities into provider registry behavior too early.
- Lets provider lifecycle, adapter, store, resolver, event bus, UI, or showcase changes sneak into a native proof slice.
- Emits private content, raw paths, command output, credentials, notification text, or transcript content.

## Stop Conditions

Stop and re-plan if the next slice needs:

- Windows API, Windows Media, GSMTC, native handles, OS observation, or background processes.
- Tauri/Rust command wiring, capability/config edits, permissions, package/dependency/script changes, or production packaging.
- Runtime-provider wiring.
- Provider adapter, store, resolver, event bus, UI, or showcase changes.
- A real provider file or exported provider path.
- Any wording that implies native Windows or Music support is already available, ready, connected, active, implemented, or production-capable.

## Recommended Next Slice

The next safest v0.8 step is another narrow planning or tests-only slice, not a native implementation slice. Prefer one of:

- A docs-only candidate decision record that chooses system status, music, or downloads as the first provider.
- A tests-only diagnostic-result shape for a future provider preflight, with no native calls and no provider export.
- A privacy checklist artifact for the selected candidate.

Real provider implementation should wait until one of those readiness steps is reviewed and approved.
