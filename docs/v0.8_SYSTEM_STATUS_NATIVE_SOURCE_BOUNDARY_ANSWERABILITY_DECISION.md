# v0.8 System Status Native Source Boundary Answerability Decision

## Purpose

This document classifies the Slice 40-A native source boundary questions by answerability. It is a docs-only answerability-planning artifact. It does not answer source questions into source behavior, APIs, native calls, implementation mechanics, runtime/provider wiring, UI behavior, config, permissions, capabilities, tests, or `windowsProviders` changes.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful and unchanged.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- This is answerability-planning only, not native/runtime/provider implementation readiness.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Decision Boundary

Answerability means reviewers can classify what kind of future evidence would be needed before a question may be answered safely. It does not mean the source question is answered here.

This artifact separates questions into three answerability classes:

- **Answerable from shipped docs/tests at boundary level**: Existing evidence can classify the question boundary, allowed vocabulary, forbidden data, or stop condition without defining source behavior.
- **Unanswered pending future evidence**: Existing evidence is not enough to answer the question safely.
- **Requires future evidence before any safe answer**: A later gate must provide additional reviewed evidence before the question can move beyond classification.

No classification in this document authorizes source calls, source behavior, Windows API calls, native handles, OS observation, polling, provider export paths, runtime-provider wiring, event emission, lifecycle behavior, UI exposure, real provider work, real toggles, permissions/capabilities, config changes, dependency changes, tests, or implementation mechanics.
