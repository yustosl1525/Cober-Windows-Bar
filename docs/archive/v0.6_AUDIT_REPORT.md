# v0.6.0 Implementation Readiness Audit

## Current Baseline

- Branch: `main`
- Latest confirmed commit: `c0fa6f7 docs: approve provider final freeze`
- v0.5.4 Final Freeze decision: `APPROVE`
- Audit mode: Reality Check / Gap Analysis only
- Implementation status: v0.6 code changes are not made in this audit

This audit inspected the current provider source and frozen planning docs to determine whether v0.6 should continue by filling gaps, refactoring existing code to the frozen contract, or replacing early experiment code with a normal implementation.

## Existing Runtime Components

Current provider-related source already exists:

- `src/providers/types.ts`
- `src/providers/mockProviders.ts`
- `src/providers/providerAdapter.ts`
- `src/providers/provider.test.ts`

The current source provides:

- a lightweight `HubProvider` interface
- four mock provider factories
- a provider-to-event-bus adapter
- basic provider and adapter tests

The existing adapter path partially matches the frozen runtime path:

```text
Provider -> Event Bus -> Store -> Resolver -> UI
```

`connectProviderToEventBus()` subscribes to provider event batches and forwards each event through `eventBus.publishHubEvent(event)`. It does not bypass the Event Bus, Store, or Resolver.

## Test Coverage Review

Current `src/providers/provider.test.ts` covers:

- music provider events resolve through the Event Bus to `music`
- AI task provider events resolve through the Event Bus to `aiProgress`
- download provider events resolve through the Event Bus to `download`
- provider starts from `stopped`
- `start()` sets provider to `running` and emits one event
- `stop()` sets provider to `stopped`
- unsubscribe prevents listener calls
- notification expiry behavior through the Event Bus
- adapter disconnect prevents forwarding
- adapter disconnect is safe to call more than once

Missing or partial test coverage versus `docs/qa/TEST_STRATEGY.md`:

- canonical lifecycle states
- health states and failure handling
- provider metadata shape
- registry registration/list/get/remove/start/stop
- duplicate id handling
- duplicate start without repeated unintended emissions
- cleanup of timers or async work after stop
- pause/resume or capability-gated pause behavior
- provider crash to Registry health transition
- canonical HubEvent contract without top-level `title`/`subtitle`
- event spam/backpressure
- progress sequence beyond a single fixed progress event
- subscription isolation when one listener throws

## Gaps

1. Contract shape drift: current `HubProvider` and `HubEvent` types are older and simpler than the frozen docs. The biggest concrete mismatch is top-level `title`/`subtitle` in `src/types/hub.ts` and mock events.
2. Lifecycle and health are not implemented as frozen. Code uses a simple `stopped/running/error` status and has no separate health model.
3. ProviderRegistry is missing. The docs describe registry-owned metadata, capabilities, lifecycle, health, and registration order, but source currently has only direct provider factories and adapter wiring.
4. Failure handling is missing. There is no code path for provider crash, Registry health marking, or optional canonical error detail in `payload`/`metadata`.
5. Existing tests protect the experimental adapter path, but do not yet cover most frozen lifecycle, registry, cleanup, duplicate start, failure, or canonical contract requirements.

## Recommended v0.6 Scope

Decision recommendation: treat v0.6 as an alignment refactor plus completion of the first mock provider SDK slice.

Minimal v0.6 scope should be:

1. Align `src/types/hub.ts` and provider event fixtures with the canonical HubEvent top-level field set while preserving Store/Resolver/UI behavior.
2. Replace or adapt `HubProviderStatus` into frozen provider lifecycle plus health status.
3. Add static provider metadata for the four mock providers without resolver/display priority.
4. Decide and implement the minimum ProviderRegistry needed for v0.6, or explicitly mark registry as a deferred gap before coding.
5. Keep the existing adapter path through `publishHubEvent()` and add contract/lifecycle validation around it.
6. Expand tests around lifecycle, health, duplicate start, unsubscribe cleanup, adapter disconnect, notification expiry, progress sequence, and canonical event shape.
7. Keep v0.6 mock-only: no Tauri, IPC, Rust, Windows APIs, real providers, system integrations, assets, package/script changes, or broad architecture expansion.

## Decision: B

`B) code/docs mismatch requires Alignment Refactor`

The existing provider source is useful and partially functional, but it is not aligned enough with the frozen v0.5.4 contract to proceed as a pure gap-fill. v0.6 should start by refactoring the current experimental provider layer to the frozen contract, then complete the minimum mock provider SDK behavior and tests.
