# v0.6.1 Alignment Refactor Plan

## Current Baseline

- Branch: `main`
- Expected HEAD: `737c62e docs: audit provider implementation readiness`
- v0.5.4 Final Freeze: `APPROVE`
- v0.6.0 Audit decision: `B) code/docs mismatch requires Alignment Refactor`
- This document is planning only. It does not implement provider code, tests, registry code, Tauri/Rust/IPC/system integration, package changes, or script changes.

Current provider source already exists:

- `src/types/hub.ts`
- `src/state/hubState.ts`
- `src/providers/types.ts`
- `src/providers/mockProviders.ts`
- `src/providers/providerAdapter.ts`

The existing implementation is useful but experimental. It already forwards provider events through `Provider -> Event Bus -> Store -> Resolver -> UI`, but its provider contract, HubEvent type, lifecycle vocabulary, health model, registry surface, and tests do not yet match the frozen docs.

## Refactor Objectives

1. Align current code with the frozen canonical HubEvent contract while preserving existing Store, Resolver, and UI behavior.
2. Replace the coarse provider status model with explicit lifecycle and health concepts.
3. Add provider metadata and mock provider records without resolver/display priority.
4. Introduce the minimum ProviderRegistry needed to own provider identity, lifecycle, health, capabilities, metadata, and deterministic registration order.
5. Expand tests around the contract gaps found in the v0.6.0 audit.
6. Keep the runtime path intact: `Provider -> Event Bus -> Store -> Resolver -> UI`.
7. Keep v0.6 mock-only and browser-local. No Tauri, IPC, Rust, Windows APIs, real providers, package changes, scripts, or assets.

## Ordered Implementation Slices

### Slice 1: Canonical HubEvent Compatibility / Alignment Plan

Goal: make `src/types/hub.ts`, provider fixtures, Store, Resolver, and tests compatible with the frozen HubEvent top-level fields.

Planned implementation steps:

1. Update `HubEvent` so canonical top-level fields are `id`, `type`, `source`, `createdAt`, optional `expiresAt`, optional `payload`, optional `progress`, and optional `metadata`.
2. Remove top-level `title` and `subtitle` from provider-authored events.
3. Keep display text inside `payload` where UI derivation currently needs it.
4. Update `eventToTask()` and related Store mapping to derive `title`, `subtitle`, and task display state from payload.
5. Preserve existing resolver outputs for music, AI, download, notification, and multi-task scenarios.
6. Add compatibility tests proving canonical events still resolve to the same hub modes.

Exit criteria:

- No provider event requires top-level `title` or `subtitle`.
- Existing showcase state resolution remains stable.
- HubEvent fields match the frozen docs.

### Slice 2: Provider Lifecycle + Health Type Alignment Plan

Goal: replace `HubProviderStatus = "stopped" | "running" | "error"` with frozen lifecycle and health vocabulary.

Planned implementation steps:

1. Define provider lifecycle values: `Registered`, `Started`, `Publishing`, `Paused`, `Stopped`, `Failed`.
2. Define provider health values: `Healthy`, `Degraded`, `Unhealthy`.
3. Replace `getStatus()` with `status()` returning a snapshot that includes lifecycle and health.
4. Make provider construction start from `Registered` or `Stopped` according to the final code shape, but keep the value explicit and tested.
5. Ensure `start()` is idempotent and does not create duplicate loops or duplicate unintended emissions.
6. Ensure `stop()` is idempotent and leaves providers in `Stopped`.
7. Decide whether `pause()` and `resume()` are implemented in v0.6 as capability-gated methods or deferred; if deferred, tests should prove unsupported pause/resume is not exposed as a fake contract.

Exit criteria:

- Code no longer uses `"stopped" | "running" | "error"` as the provider contract.
- Lifecycle and health are separate.
- Failure state can be represented without mixing lifecycle, health, and event/task payload status.

### Slice 3: Provider Metadata + Mock Provider Alignment Plan

Goal: align the four mock providers with frozen metadata and naming expectations.

Planned implementation steps:

1. Add static provider metadata to each mock provider: stable `id`, `name`, `kind`, `version`, and `mock`.
2. Add capability declarations for the four v0.6 providers: music, AI, download, and notification.
3. Keep provider metadata free of resolver priority, display priority, final hub mode, UI state, or ranking hints.
4. Rename or alias `createMockAiTaskProvider` toward the canonical `MockAIProvider` naming while minimizing churn.
5. Keep mock event sequences deterministic through injected `now` or deterministic fixture options.
6. Keep mock providers as HubEvent producers only; they should not render UI, mutate Store directly, or decide Resolver priority.

Exit criteria:

- Music, AI, Download, and Notification mock providers have consistent metadata and capability shape.
- Provider ids are stable and suitable for registry registration.
- No metadata field influences Resolver priority.

### Slice 4: Minimum ProviderRegistry Plan

Decision: implement a minimum ProviderRegistry in v0.6 rather than deferring it.

Rationale:

- The audit identified registry absence as a core gap, not a nice-to-have.
- Frozen docs assign lifecycle, health, metadata, capabilities, and deterministic registration order to the registry.
- Failure handling depends on a source of truth for provider health.
- Tests need a registry surface to verify duplicate ids, lookup, list order, and lifecycle state ownership.

Planned minimum registry scope:

1. `register(provider)` stores one provider by stable id and assigns deterministic registration order.
2. Duplicate ids are rejected and leave the existing provider unchanged.
3. `get(providerId)` returns the registered provider record or an empty result.
4. `list()` returns providers in registration order.
5. `unregister(providerId)` removes a provider and disconnects or stops it if needed.
6. `start(providerId)` and `stop(providerId)` delegate lifecycle calls and update registry-visible lifecycle/health.
7. `startAll()` and `stopAll()` may be included only if they are tiny wrappers over the single-provider operations.

Deferred registry scope:

- persistence
- settings UI
- real provider discovery
- permission prompts
- network or OS availability checks
- Tauri/native process supervision
- complex filtering beyond the four mock provider groups

Exit criteria:

- Registry can register, list, get, reject duplicate ids, start, stop, and unregister the four mock providers.
- Registry does not bypass Event Bus, Store, or Resolver.
- Registry owns health/lifecycle visibility, not final UI state.

### Slice 5: Test Expansion Plan

Goal: convert audit gaps into executable protection before broadening behavior.

Minimum test additions:

1. Canonical HubEvent tests: provider events do not require top-level `title` or `subtitle`; display text is payload-derived.
2. Lifecycle tests: initial state, start, duplicate start, stop, duplicate stop, and failure representation.
3. Health tests: healthy default, unhealthy on simulated failure, and lifecycle/health separation.
4. Metadata tests: four providers expose stable metadata and no resolver/display priority fields.
5. Registry tests: register, list, get, duplicate id rejection, unregister, start, stop, and registration order.
6. Adapter tests: provider events still flow through `publishHubEvent()` and disconnect remains idempotent.
7. Cleanup tests: unsubscribe prevents later emissions and stop prevents post-stop event delivery.
8. Notification expiry tests: existing expiry behavior remains intact after canonical HubEvent alignment.
9. Progress sequence tests: AI/download progress remains deterministic and monotonic for the mock fixtures.

Exit criteria:

- Existing provider tests are updated rather than discarded blindly.
- Tests fail on the audit gaps before implementation fixes them where practical.
- `npm run qa` passes after the implementation slices are complete.

## Risk / Compatibility Notes

- Removing top-level `title` and `subtitle` from `HubEvent` can break Store/UI assumptions. Mitigate by deriving view data from payload inside Store mapping and protecting existing modes with tests.
- Provider lifecycle names are PascalCase in docs. Changing from lowercase status strings will require test updates and possibly call-site updates.
- Duplicate `start()` currently re-emits events. The implementation must define whether a second start is a no-op or returns the existing active state without unintended emissions.
- Registry implementation can expand quickly. Keep it in-memory, deterministic, and provider-local for v0.6.
- Failure handling should not invent a new top-level failure event type. Use registry health plus optional canonical HubEvent payload/metadata when user-visible.
- Mock AI naming should settle on `MockAIProvider` while preserving compatibility where cheap.
- The existing UI may still need display-ready fields. Keep UI derivation downstream of Store/Resolver rather than pushing UI fields back into provider events.

- any Tauri, Rust, IPC, Windows API, or system integration becomes necessary
- existing source shows a larger architectural conflict than the v0.6.0 audit identified
- provider failure handling cannot be represented with registry health plus canonical HubEvent payload/metadata

Recommendation: GO for a narrow v0.6 alignment refactor only after this plan is accepted. Do not start implementation from this document alone without monitor approval.
