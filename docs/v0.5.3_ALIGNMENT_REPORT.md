## Five Alignment Decisions

### 1. HubEvent Contract

The canonical v0.6 `HubEvent` top-level fields are:

- `id`
- `type`
- `source`
- `createdAt`
- optional `expiresAt`
- optional `payload`
- optional `progress`
- optional `metadata`

Display text, task status, provider-specific phase labels, priority, and updated timestamps are not canonical top-level event contract fields. They belong in `payload`, `metadata`, or future extensions after the first mock provider slice is proven.

### 2. Runtime Path

The canonical runtime path is:

```text
Provider -> Event Bus -> Store -> Resolver -> UI
```

Provider registry and provider adapter layers may support metadata, health, lifecycle supervision, and event forwarding, but they must not bypass the Event Bus, Store, Resolver, or UI boundary.

### 3. Lifecycle, Health, and Event Status

Provider lifecycle, registry health, and event/task status are separate concepts.

- Provider lifecycle: `Registered`, `Started`, `Publishing`, `Paused`, `Stopped`, `Failed`
- Registry health: `Healthy`, `Degraded`, `Unhealthy`
- Event/task status: provider-specific task detail carried in `payload` or `metadata`

The lifecycle vocabulary should not be reused as event status, and event/task status should not drive provider health.

### 4. Registry Matrix

The registry must know each provider's id/name/group, capabilities, lifecycle state, health, metadata, and registration order. Provider-specific implementation details remain private to the provider.

The v0.6 registry matrix is limited to:

- `MockMusicProvider`
- `MockAIProvider`
- `MockDownloadProvider`
- `MockNotificationProvider`

### 5. v0.6 Scope

v0.6 may implement the first mock provider runtime slice only after v0.5.4 approves the aligned docs.

Allowed v0.6 implementation targets:

- `ProviderRegistry`
- `MockMusicProvider`
- `MockAIProvider`
- `MockDownloadProvider`
- `MockNotificationProvider`

Forbidden v0.6 scope:

- Tauri
- Rust / IPC
- Windows APIs
- Real providers
- Music Session
- Notification Center
- Downloads API
- System integrations

## v0.5.2 Blockers Resolved

- HubEvent contract drift was aligned to one canonical field set.
- Runtime path wording was aligned to one main path.
- Provider lifecycle, registry health, and event/task status were separated.
- Registry knowledge and provider-private boundaries were clarified.
- Mock provider event examples now use canonical fields.
- v0.6 scope is described as mock-only and remains blocked until v0.5.4 review.

## Remaining Open Question
