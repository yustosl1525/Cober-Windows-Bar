# v0.8 System Status Native/Runtime/Provider Path Plan

## Purpose

This document defines the future native/runtime/provider path at sequencing, contract, and gate level only. It is a docs-only planning artifact. It does not authorize implementation.

This path plan does not define code APIs, Rust/Tauri commands, provider exports, runtime-provider wiring, event behavior, UI behavior, permissions, capabilities, config toggles, polling, native handles, Windows API behavior, or implementation mechanics.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful and unchanged.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- This is planning-path readiness only, not native/runtime/provider implementation readiness.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Path Decision

The shipped v0.8 evidence chain is complete enough to plan the next native/runtime/provider path. It is not complete enough to start native/runtime/provider implementation.

The next path should proceed only through separate future gates. Any real work must receive a future PLAN gate, implementation gate, review gate, commit gate, and push gate before it lands. This document does not start that work.

## Rollback / Disable / Kill-Switch Ties

Every future phase must keep the Slice 36-A criteria active:

- Roll back to the last approved docs-only or tests-only boundary if a phase needs private/raw data.
- Keep the path disabled if any required evidence gate is missing.
- Use kill-switch decision inputs only as reviewable, non-private criteria.
- Keep diagnostics bounded, redacted, and facts-only.
- Keep `windowsProviders: false` unless a later separately approved runtime capability implementation gate proves and changes it.
- Stop if a phase crosses into unapproved native collection, provider lifecycle, runtime-provider wiring, event emission, UI exposure, real toggles, permissions, capabilities, config, package, dependency, or native API work.

## Phase Order

The path should move in this order unless a later PLAN-only gate changes sequencing:

| Phase | Purpose | Required approval before proceeding | Must remain outside the phase |
| --- | --- | --- | --- |
| Phase 0: Standby baseline | Preserve the shipped evidence chain and current runtime facts. | No approval needed to remain in standby. | Any implementation, `windowsProviders` change, or native/system status support claim. |
| Phase 1: First concrete phase plan | Choose the first narrow future phase and define its review boundary. | Separate PLAN-only gate. | Code, tests, runtime/provider/native changes, and implementation mechanics. |
| Phase 2: Local contract proof, if needed | Add or revise local fixture coverage only if Phase 1 identifies a concrete gap. | Separate tests-only implementation gate after review. | Production behavior, native collection, provider export, runtime-provider wiring, event emission, and UI exposure. |
| Phase 3: Native source boundary plan | Define source-boundary questions and failure-mode expectations without source calls. | Separate docs-only PLAN and implementation gates. | Windows API calls, native handles, OS observation, polling, Rust/Tauri commands, permissions, capabilities, config, and dependencies. |
| Phase 4: Runtime capability transition plan | Reassess whether any future `windowsProviders` proposal has enough evidence. | Separate PLAN-only gate, then any implementation gate if later approved. | Changing `windowsProviders`, runtime-provider wiring, provider lifecycle, event emission, or UI exposure in the planning slice. |
| Phase 5: Provider path plan | Define how provider lifecycle would remain separate from diagnostics and runtime payload facts. | Separate PLAN-only gate. | Provider exports, provider files, provider startup, provider subscription, event emission, lifecycle behavior, and real provider work. |

These phases are dependency order only. They do not create implementation authorization.

## Future Approval Gates

Any future real work must pass these gates in order:

1. Commander opens a narrow PLAN-only checkpoint.
2. Tester approves the plan.
3. Commander authorizes implementation-only scope.
4. Tester reviews implementation.
5. Commander authorizes commit-only scope.
6. Tester reviews the commit.
7. Commander authorizes push-only scope.
8. Tester performs post-push review before closure.

Any future CodeGraph sync requires separate authorization. This document authorizes no CodeGraph sync.

## Future Test Expectations

Future tests, if separately approved, should prove only the boundary under review:

- Diagnostic tests should keep unsupported platform, source unavailable, permission denied, malformed source data, timeout, and provider bug cases bounded.
- Runtime payload tests should keep CPU range, memory pressure, battery state, and network availability summary coarse and redacted.
- Privacy regression tests should reject secrets, paths, command output, usernames, app identities, raw network data, stable identifiers, raw OS data, native handles, and other private/raw content.
- Runtime capability tests should keep `windowsProviders: false` truthful until a later approved runtime capability implementation changes it.
- Provider separation tests should reject provider lifecycle, provider export, runtime-provider wiring, event emission, Store/Resolver/UI, polling, and native collection fields unless a later gate explicitly approves a different scope.

This slice changes no tests.
