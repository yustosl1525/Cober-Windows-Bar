# Cober-Windows-Bar Event Flow

This document describes the event flow architecture as of the Tauri integration (post-v0.7). Events flow from both mock and native sources through a unified pipeline to the UI.

## Canonical HubEvent Contract

`HubEvent` is the normalized status message passed from providers into the hub event path.

```ts
type HubEventType =
  | "music"
  | "ai"
  | "download"
  | "notification"
  | "system"
  | "developer";

interface HubEvent {
  id: string;
  type: HubEventType;
  source: string;
  createdAt: number;
  expiresAt?: number;
  progress?: number;
  payload?: Record<string, unknown>;
  metadata?: Record<string, unknown>;
}
```

Design notes:

- `id` gives the Store a stable key for replace/update behavior.
- `type` lets the Resolver map events to hub modes.
- `source` identifies the provider, runtime, or mock scenario that emitted the event.
- `createdAt` is the canonical event timestamp.
- `expiresAt` is optional and lets the Store expire time-bound events.
- `progress` is optional and should be clamped from `0` to `100` when present.
- `payload` is the event-specific data envelope. Provider-specific task status, display copy, and domain details belong here unless a future version promotes them.
- `metadata` is optional descriptive metadata. It is not a resolver contract unless a later version explicitly says so.

Non-canonical top-level fields:

- `kind` is replaced by canonical `type`.
- Event-level `status` is not canonical. Task status such as active, progress, complete, paused, error, or cleared belongs in `payload` unless a future contract promotes it.
- `title` and `subtitle` are display payload, not top-level contract fields.
- `priority` is a future resolver concern, not a canonical event field.
- `updatedAt` is not part of the frozen top-level contract.

## Event Sources

Events can originate from three sources, selected by the runtime bridge layer:

1. **Mock providers** (`src/providers/mockProviders.ts`) — deterministic fake events for development and testing, usable without Tauri.
2. **Tauri fixture events** — synthetic events generated by the Rust backend (`get_hub_event_fixtures` / `emit_hub_event_fixtures`), useful for IPC boundary testing.
3. **Tauri native events** — real system data from Rust commands, pushed to the frontend via Tauri's event system (`status-center://hub-events`).

The `desktopStatusInputRuntime` manages source selection with a three-tier fallback:

```text
Mock → Tauri Fixture → Tauri Event Push
```

## Event Bus Semantics

The Event Bus (`createHubEventBus` in `src/state/hubState.ts`) is the transport boundary between providers/runtime and hub state.

Responsibilities:

- Accept published `HubEvent` objects.
- Notify subscribers in publication order.
- Keep provider adapters decoupled from store internals.
- Remain synchronous and deterministic.
- Snapshot events at the transport boundary so caller-side mutations cannot change stored event payloads or metadata after handoff.
- Isolate adapter publish failures per event (one failed publish does not block others in the same batch).

Canonical runtime path:

```text
Provider/Runtime -> Event Bus -> Store -> Resolver -> UI
```

## Store Responsibilities

The Store (`createHubStoreState` in `src/state/hubState.ts`) owns the active event snapshot used by the resolver.

Responsibilities:

- Insert new events by `id`.
- Replace or merge updates for an existing `id`.
- Remove events when a canonical event operation or payload-level task state indicates clearing.
- Expire events when `expiresAt` is in the past.
- Clear all events for Idle/demo reset flows.
- Expose active events to the Resolver and showcase diagnostics.
- Return Store snapshots as copied read models so callers cannot mutate stored events, payloads, metadata, music state, or notification state through a resolved snapshot.

The Store does not know how music sessions, downloads, notifications, developer tools, or AI agents are collected. Those details belong to providers.

## Resolver Rules

The Resolver (`resolveDesktopStatusState` in `src/state/desktopStatusState.ts`) turns the Store snapshot into one visible hub mode.

Responsibilities:

- Apply deterministic priority rules.
- Resolve one primary mode for the compact hub.
- Return MultiTask when multiple meaningful events compete.
- Return Idle when no active event remains.
- Consider attention reasons (new, near-complete, completion, urgent).
- Respect preferred-kind windows and user interaction timing.

Priority order:

1. Notification (urgent interruption)
2. MultiTask (multiple active meaningful events)
3. AI Progress (long-running agent work)
4. Download (file transfer progress)
5. Music (ongoing media)
6. Resident (system performance when nothing else active)
7. Idle (no active events)

Deterministic rules:

- Ignore expired events before resolving.
- Ignore events cleared by Store behavior.
- If no active events remain, resolve Idle.
- If two or more active events remain after filtering, resolve MultiTask unless one event has explicit interruption priority.
- Notifications may temporarily override MultiTask when resolver rules or payload-level data mark them as interruptions.
- For events of the same `type`, prefer deterministic resolver rules and lexical `id` as the final tie-breaker.
- The same Store snapshot always produces the same resolver output.

### Scheduler Logic

The `desktopStatusScheduler` implements:

- **Preferred-kind pinning**: A user-selected kind stays active for a configurable window (`DESKTOP_STATUS_PREFERRED_WINDOW_MS`).
- **User interaction lock**: After user interaction, the current mode is held for a grace period.
- **Guest cooldown**: Guest status kinds (non-resident) observe a cooldown period before yielding to lower-priority states.
- **Attention tracking**: Events marked as "new", "near-complete", "completion", or "urgent" receive temporary priority boosts.

## Desktop Page Integration

The `DesktopPage.tsx` component orchestrates the full event flow:

1. **Initialization**: Creates a `DesktopStatusRuntime` instance that manages the three-tier source selection.
2. **Polling loop**: Every 1800ms, fetches system performance metrics, guest provider capabilities, and media session status from Tauri IPC.
3. **Event subscription**: Subscribes to runtime state changes and updates the hub store.
4. **Overlay policy**: Every 700ms, checks fullscreen state and adjusts window visibility.
5. **Display/scale monitoring**: Listens for window move, resize, and DPI scale change events with debounced position correction.
6. **Menu/settings bridge**: Listens for Tauri events for context menu actions, settings changes, and settings-open requests.

## Example Flow

```text
Idle -> Music -> AI -> Notification -> MultiTask -> Idle
```

1. **Idle**: Store has no active events. Resolver returns `Idle`. UI renders `ResidentStatusTemplate` with system metrics.

2. **Music starts**: Media session provider detects playback via GSMTC. Event published through Tauri IPC → runtime adapter → event bus → store. Resolver returns `Media`. UI renders `MediaStatusTemplate`.

3. **AI task starts**: An AI agent provider publishes an `ai` event with optional `progress`. Store now has music and AI events. Resolver returns `MultiTask`.

4. **Notification arrives**: A notification provider publishes a `notification` event. Resolver treats it as highest-priority interruption. UI renders notification template.

5. **Notification settles**: Notification expires. Resolver returns `MultiTask` because music and AI remain active.

6. **Clear to Idle**: All events expire or are cleared. Resolver returns `Idle`. UI returns to resident status view.
