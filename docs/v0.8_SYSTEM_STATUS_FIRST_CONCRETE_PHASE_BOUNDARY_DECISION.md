# v0.8 System Status First Concrete Phase Boundary Decision

## Purpose

This document selects the first concrete future phase for system status work at boundary, decision, and gate level only. It is a docs-only planning artifact.

This decision does not define code APIs, module paths, Rust/Tauri commands, provider exports, runtime-provider wiring, events, UI exposure, permissions, capabilities, config toggles, polling, native handles, Windows API behavior, lifecycle behavior, or implementation mechanics.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful and unchanged.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- This is first-phase planning readiness only, not native/runtime/provider implementation readiness.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Boundary Decision

The first concrete future phase should be docs-only native source boundary questions.

This phase should define what reviewers need to know before any future source-boundary work can be considered. It must not call a source, observe the operating system, create native handles, start polling, define Rust/Tauri commands, create provider export paths, wire runtime-provider behavior, emit events, change UI exposure, add permissions/capabilities, add config toggles, or change `windowsProviders: false`.

Tests-only local contract proof remains deferred until a concrete gap is identified. The current local fixture coverage already proves the planned diagnostic and runtime payload boundaries for the current evidence set.

## Owner Boundary

The first concrete phase has these owner boundaries:

- Commander owns whether to open a later PLAN-only checkpoint for native source boundary questions.
- Tester owns read-only review of that future plan.
- Developer owns stopping if the scope crosses from boundary questions into implementation mechanics.
- Any later implementation requires a separate implementation-only authorization.
- Any later commit, push, or CodeGraph sync requires separate authorization.

This decision authorizes no source calls and no implementation.

## Native Source Boundary Questions

- Which coarse system status facts remain useful without private/raw data?

These are questions for planning only. They do not define source behavior, API calls, provider lifecycle, runtime payload implementation, event emission, or UI behavior.
