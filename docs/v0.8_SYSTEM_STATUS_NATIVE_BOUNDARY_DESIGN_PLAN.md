# v0.8 System Status Native Boundary Design Plan

## Purpose

This document defines the future native boundary design for the system status capability. It is a docs-only readiness artifact. It does not implement provider behavior, call a Windows API, observe the operating system, start polling, create a provider export path, emit events, change lifecycle behavior, wire runtime capabilities into provider behavior, or expose native data to UI.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Native Boundary Intent

The future native boundary should isolate source collection from the rest of the application. Its first job is not to publish product behavior. Its first job is to prove that native source outcomes can be converted into bounded diagnostics and planned runtime payload facts without leaking private data or crossing provider lifecycle boundaries.

The boundary may be designed around this future flow:

```text
future native source -> redaction boundary -> diagnostic result -> runtime payload plan
```

It must not become this flow without a later approved implementation slice:

```text
native source -> provider lifecycle -> event emission -> Store/Resolver/UI
```

## Boundary Responsibilities

| Responsibility | Future boundary may define | Must remain outside this slice |
| --- | --- | --- |
| Source collection | Which coarse system status facts would be requested later. | Windows API calls, native handles, OS observation, polling implementation, or background processes. |
| Redaction | How raw source outcomes would be reduced before leaving the boundary. | Raw process lists, paths, command output, app identities, raw network details, secrets, or identifiers. |
| Diagnostics | How unsupported platform, source unavailable, permission denied, malformed source data, timeout, and provider bug cases would be summarized. | Direct Store, Resolver, UI, provider lifecycle, or event emission behavior. |
| Runtime payload planning | What future runtime payload contract questions must be answered before `get_runtime_capabilities` changes. | Any `windowsProviders` value change or runtime-provider wiring. |
| Provider relationship | How provider facts stay separate from runtime capability facts. | Provider export paths, provider registration, startup, subscription, or lifecycle behavior. |
| UI exposure | Which coarse facts might eventually be safe for UI after later review. | Direct UI wiring, showcase changes, or HubEvent publication. |

## Allowed Future Boundary Facts

The boundary design may discuss only future coarse facts already allowed by the privacy checklist:

- CPU load range.
- Memory pressure category.
- Battery state.
- Network availability summary.

These facts must remain derived, bounded, and coarse. Any future design must summarize them before they cross into diagnostics, runtime payload planning, HubEvents, logs, or tests.

The boundary design may also reference:

- `windowsProviders: false` as the current truthful runtime capability fact.
- `origin: "native"` and `support: "preflight"` as diagnostic metadata only.
- The `system-status` descriptor id and kind as capability facts only.
- The six redacted diagnostic categories proven by local fixtures.

## Forbidden Boundary Behaviors

The native boundary design must not include or imply:

- Windows API calls, native handles, OS observation, background processes, or polling implementation.
- Tauri/Rust command wiring, permissions, capabilities, config, package, dependency, or script changes.
- Provider export paths, provider files, provider startup behavior, provider subscription behavior, provider lifecycle behavior, or provider availability.
- Runtime-provider wiring or a `windowsProviders` change.
- Event emission, HubEvent payload publication, Store mutation, Resolver behavior, UI exposure, or showcase behavior.
- Raw process lists, window titles, command output, usernames, paths, app identities, notification text, transcript content, credentials, raw network details, or stable identifiers.
- Wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Evidence Required Before Later Implementation

A later implementation plan should not begin until the evidence package is complete:

1. Privacy checklist satisfied for every proposed coarse fact.
2. Descriptor boundary satisfied with `origin: "native"` and `support: "preflight"` still diagnostic-only.
3. Redacted diagnostics proven for unsupported platform, source unavailable, permission denied, malformed source data, timeout, and provider bug cases.
4. Native boundary design reviewed without approving native calls or OS observation.
5. Failure modes defined so native failures stay inside diagnostics.
6. Runtime payload contract planned before any `get_runtime_capabilities` or `windowsProviders` transition.
7. Separate implementation authorization granted before any native/runtime/provider work begins.

If any item is missing, the next slice should remain docs-only or tests-only and keep `windowsProviders: false`.

## Separation Rules

### Future Collection

Future collection describes what a later source boundary might request. It must not call native APIs, observe the operating system, start background work, or create a provider path in this slice.

### Redaction

Redaction must happen before any fact leaves the future native boundary. Raw source values must be reduced to enum, boolean, or short-code facts before diagnostics or runtime payload planning use them.

### Runtime Payload Planning

Runtime payload planning describes how a later `get_runtime_capabilities` payload might represent evidence. It is separate from provider behavior and must not change `windowsProviders: false` in this slice.

### Provider Lifecycle

Provider lifecycle remains out of scope. A boundary diagnostic must not imply a provider is registered, started, subscribed, publishing, stopped, failed, or otherwise operating.

### Event Emission

Event emission remains out of scope. A boundary diagnostic must not publish HubEvents or update Store, Resolver, UI, or showcase state.

### UI Exposure

UI exposure remains out of scope. Any future UI-facing system status fact must wait for a later approved payload and event design.

## Relationship To Existing Gates

This design preserves the existing v0.8 readiness gates:

- The privacy checklist owns allowed coarse facts and forbidden private data.
- The preflight descriptor plan owns diagnostic capability metadata.
- The Slice 29-A diagnostic categories remain local fixture categories until a later slice proves otherwise.
- The runtime capability transition evidence plan owns the evidence gate before any `windowsProviders` proposal.

This document does not replace those gates. It connects them so a future native boundary proposal can be reviewed without starting implementation.

## Review Criteria

A future native boundary proposal should be accepted only if it:

- Keeps `windowsProviders: false` truthful until a later approved runtime capability implementation changes it.
- Keeps `origin: "native"` and `support: "preflight"` diagnostic-only.
- Separates native collection, redaction, runtime payload planning, provider lifecycle, event emission, and UI exposure.
- Preserves the privacy checklist, descriptor boundary, redacted diagnostic categories, and transition evidence gates.
- Requires separate implementation authorization before any native calls, OS observation, polling, provider export path, runtime-provider wiring, or event emission.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Code, tests, config, package, dependency, script, Rust, Tauri, runtime, provider, store, resolver, event bus, UI, or showcase changes outside its approved scope.
- Windows API calls, native handles, OS observation, polling implementation, or background processes.
- Tauri/Rust command wiring, permissions, capabilities, config, package, dependency, or script changes.
- Provider export paths, real provider files, provider startup behavior, provider subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring or a `windowsProviders` value change.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Next Sequencing

The next safe follow-up remains non-native. Prefer a docs-only runtime payload contract plan or a tests-only local fixture contract after this boundary design is reviewed. Real system status provider work should wait until privacy, descriptor, diagnostic, transition evidence, native boundary, and runtime payload gates are each separately reviewed and approved.
