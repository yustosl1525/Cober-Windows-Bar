# v0.8 System Status Privacy Checklist

## Purpose

This checklist defines the privacy boundary for a future system status provider candidate. It is a docs-only readiness artifact. It does not implement provider behavior, call a Windows API, observe the operating system, start polling, create a provider export path, emit events, or wire runtime capabilities into provider behavior.

The current baseline remains unchanged:

- `windowsProviders: false` remains truthful.
- Provider `origin: "native"` with `support: "preflight"` remains diagnostic-only.
- Native system status support is not available, ready, connected, active, implemented, production-capable, or already wired.

## Allowed Coarse Facts

A future system status candidate may only propose derived, coarse facts until a later approved implementation slice narrows the source boundary.

Allowed fact categories:

| Fact | Allowed shape | Privacy rule |
| --- | --- | --- |
| CPU load range | Low, medium, high, or critical range. | Do not expose process names, process ids, command lines, window titles, or per-app usage. |
| Memory pressure category | Normal, elevated, high, or critical pressure. | Do not expose process lists, app identities, usernames, paths, or memory maps. |
| Battery state | Charging, discharging, full, low, critical, or unknown. | Do not expose device serials, account identifiers, hardware ids, or power-plan names. |
| Network availability summary | Offline, online, limited, metered, or unknown. | Do not expose SSIDs, IP addresses, MAC addresses, hostnames, remote endpoints, raw network details, or traffic contents. |

Allowed facts should be summarized before they cross into `HubEvent.payload`, `HubEvent.metadata`, logs, diagnostics, or test fixtures.

## Forbidden Facts

The system status candidate must not read, store, log, emit, or test-normalize these facts without a separate privacy review and explicit implementation scope:

- Raw process lists.
- Process ids, executable names, command lines, or command output.
- Window titles.
- Usernames, home directories, account names, or user identifiers.
- File paths, folder paths, project paths, downloads paths, or drive contents.
- App identities, package identities, app usage history, or foreground app names.
- Notification text, notification metadata, or notification center contents.
- Transcript content, AI session text, chat logs, prompts, or generated output.
- Credentials, tokens, secrets, cookies, environment variables, or config values.
- Raw network details, including SSIDs, IP addresses, MAC addresses, hostnames, remote endpoints, packet data, DNS queries, or traffic contents.
- Hardware serial numbers, device identifiers, or stable machine identifiers.

If a future design cannot provide useful status without one of these facts, it must stop and re-plan before implementation.

## Error Summary Rules

Diagnostics should summarize failure categories without leaking private data. Error details should be stable enough for tests and review, but they must not include secrets, paths, raw content, command output, raw network details, usernames, app identities, or machine identifiers.

| Diagnostic category | Allowed summary | Redaction rule |
| --- | --- | --- |
| `unsupported-platform` | Platform is outside the approved provider boundary. | Do not include OS build strings, device identifiers, or environment dumps. |
| `source-unavailable` | Source could not be reached or reported no coarse facts. | Do not include source handles, device names, app identities, paths, or raw API responses. |
| `permission-denied` | Required future permission was not granted. | Do not include account names, policy details, registry paths, or permission prompt text. |
| `malformed-source-data` | Source data did not match the expected coarse shape. | Do not include raw source payloads, command output, process data, or network data. |
| `timeout` | Source did not return within the planned time boundary. | Do not include stack dumps, source handles, hostnames, paths, or command output. |
| `provider-bug` | Provider logic failed before producing safe coarse facts. | Do not include stack traces with paths, secrets, usernames, raw content, or identifiers. |

Preferred diagnostic payloads should use short codes, bounded messages, and boolean or enum facts. Free-form detail should be treated as risky until a later slice defines a redaction policy.

## HubEvent Boundary

If a later slice proposes converting coarse system status facts into `HubEvent` data, it must keep the event payload small and derived.

Allowed payload direction:

```text
system status diagnostic -> coarse fact enum -> canonical HubEvent -> Event Bus -> Store -> Resolver -> UI
```

Forbidden payload direction:

```text
raw OS data -> raw provider payload -> Store/UI
```

The provider must not bypass the Event Bus, Store, or Resolver. This checklist does not approve event emission, lifecycle behavior, provider startup, provider subscription, or UI behavior.

## Runtime Capability Boundary

This checklist does not change runtime capability facts.

- `windowsProviders: false` stays truthful until a later runtime capability slice proves and reviews a different value.
- `origin: "native"` with `support: "preflight"` stays diagnostic-only.
- A future system status checklist, descriptor, or test fixture must not claim native support is available, ready, connected, active, implemented, production-capable, or already wired.

## Stop Conditions

Stop and re-plan if a future slice needs:

- Windows API calls, native handles, OS observation, system polling implementation, or background processes.
- Tauri/Rust command wiring, permissions, capabilities, config, package, dependency, or script changes.
- Provider export paths, real provider files, provider startup behavior, subscription behavior, event emission, or lifecycle behavior.
- Runtime-provider wiring.
- Provider adapter, store, resolver, event bus, UI, or showcase changes.
- Raw process lists, window titles, command output, usernames, paths, app identities, notification text, transcript content, credentials, raw network details, or identifiers.
- Any wording that implies native system status support is available, ready, connected, active, implemented, production-capable, or already wired.

## Recommended Next Slice

After this checklist is reviewed, the next safe step should remain non-native:

- A tests-only diagnostic result shape slice using local fixtures only.
- A docs-only preflight descriptor plan for system status capability facts.
- A docs-only runtime capability transition plan describing the evidence required before `windowsProviders` could truthfully change.

Real system status provider implementation should wait until those readiness slices are separately reviewed and approved.
